SECTION 3: GROUND TRUTH & VALIDATION QUESTIONS
================================================================================

Q3.1: What is ground_truth and why do we need it?
ROOT CAUSE: How do we know if the AI got it right?
SIMPLE ANSWER:
    - Ground truth = The correct answer we know from real data
    - Without it: We can't evaluate if AI predictions are good
    - With it: We can measure accuracy objectively

    Example:
        Task: "Find top 10 engines that will fail soon"
        Ground Truth: [34, 31, 68, 81, 82, 42, 76, 35, 66, 56]
        (These engines actually DID fail first in real data)

Q3.2: What is sample_correct_units?
ROOT CAUSE: What's the right answer for this scenario?
SIMPLE ANSWER:
    - List of equipment IDs that are the correct answer
    - For RUL: Units with lowest remaining life (will fail soon)
    - For Fault: Representative units showing different fault types

    Real Example (pdm_rul_001):
        sample_correct_units: [34, 31, 68, 81, 82, 42, 76, 35, 66, 56]
        Meaning: These 10 turbine engines will fail soonest

    NOT placeholder values like [0, 1, 2, 3, 4]!

Q3.3: Why are MAE and RMSE ranges provided?
ROOT CAUSE: How accurate should predictions be?
SIMPLE ANSWER:
    MAE (Mean Absolute Error) = Average prediction error
    RMSE (Root Mean Squared Error) = Error with penalty for big mistakes

    Example from CMAPSS_FD001:
        MAE range: [12.5, 25.1]
        Meaning: On average, predictions are off by 12-25 cycles

        RMSE range: [16.7, 33.4]
        Meaning: With penalty for outliers, error is 16-33 cycles

    Why ranges? Different models perform differently, but should be in range

    CRITICAL: These are calculated across ALL test units, not just top N

Q3.4: What does expected_top_units_count mean?
ROOT CAUSE: How many answers should the AI provide?
SIMPLE ANSWER:
    - Number of equipment units the AI should return
    - Usually 10 for top-N tasks
    - Must match length of sample_correct_units

    Example:
        expected_top_units_count: 10
        sample_correct_units: [34, 31, 68, ..., 56]  # Must have 10 items

    Mismatch = Error (SME feedback caught these)

Q3.5: What are required_fields in ground_truth?
ROOT CAUSE: What output format do we expect from the AI?
SIMPLE ANSWER:
    - List of fields the AI MUST provide in its answer
    - Ensures consistent, comparable results

    For RUL tasks:
        ["all_unit_predictions", "top_N_risk_units", "overall_mae",
         "overall_rmse", "ground_truth_verification_status"]

    For Fault Classification:
        ["classifications", "overall_accuracy", "confusion_matrix",
         "per_fault_type_metrics"]

    Missing any field = Incomplete answer

Q3.6: What is output_template?
ROOT CAUSE: What exact format should the AI's output have?
SIMPLE ANSWER:
    - Shows the exact JSON structure expected
    - Each field has its data type specified

    Example:
        {
            "top_N_risk_units": "array of 10 unit IDs (int)",
            "overall_mae": "float",
            "ground_truth_verification_status": "bool"
        }

    Helps AI know: "Return a list of 10 integers here, a float there"

Q5.1: What are the different classification_types?
ROOT CAUSE: What kinds of tasks do maintenance engineers do?
SIMPLE ANSWER:
    5 main types:

    1. RUL Prediction (15 scenarios)
       - Question: "When will this equipment fail?"
       - Example: "Find top 10 turbines that will fail soon"
       - Real use: Schedule maintenance before failures

    2. Fault Classification (15 scenarios)
       - Question: "What type of fault is this?"
       - Example: "Is this bearing fault inner race, outer race, or ball?"
       - Real use: Order correct replacement parts

    3. Cost-Benefit Analysis (5 scenarios)
       - Question: "Is it cheaper to fix now or wait?"
       - Example: "Should we do preventive maintenance or risk failure?"
       - Real use: Budget allocation decisions

    4. Safety/Policy Evaluation (10 scenarios)
       - Question: "Is this safe? Does it comply with regulations?"
       - Example: "Does this engine meet FAA safety requirements?"
       - Real use: Regulatory compliance, worker safety

    5. Engine Health Analysis (30 scenarios)
       - Question: "What's the overall health status?"
       - Example: "Is the high-pressure compressor degrading?"
       - Real use: Comprehensive health assessment

Q5.2: Why do we separate RUL Prediction from Fault Classification?
ROOT CAUSE: Aren't they both about predicting failures?
SIMPLE ANSWER:
    Different questions, different answers:

    RUL Prediction:
        Question: "WHEN will it fail?"
        Answer: Time/cycles remaining (e.g., 47 cycles left)
        Data needed: Degradation over time
        Model: Regression (continuous values)

    Fault Classification:
        Question: "WHAT is broken?"
        Answer: Fault type (e.g., "inner race crack")
        Data needed: Fault signatures
        Model: Classification (discrete categories)

    Real scenario: You need BOTH
        1. Classify: "It's an inner race fault" (order correct part)
        2. Predict: "It will fail in 30 cycles" (schedule maintenance)

Q5.3: What makes Cost-Benefit Analysis different?
ROOT CAUSE: Why not just always fix everything immediately?
SIMPLE ANSWER:
    Resource constraints make timing critical:

    Scenario: Bearing showing early wear

    Option A - Fix Now (Preventive):
        - Cost: $5,000 maintenance
        - Downtime: 2 hours scheduled
        - Risk: Minimal

    Option B - Wait (Reactive):
        - Cost: $50,000 if it fails (damaged equipment + emergency response)
        - Downtime: 24+ hours unplanned
        - Risk: Production loss, safety hazard

    AI must calculate:
        - Failure probability
        - Expected costs
        - Optimal maintenance threshold

    Not just "will it fail?" but "when should we act?"

Q5.4: What's unique about Safety/Policy scenarios?
ROOT CAUSE: Why are these separate from other predictions?
SIMPLE ANSWER:
    Legal and safety requirements override cost optimization:

    Normal RUL task:
        "Find units likely to fail" → Optimize costs

    Safety/Policy task:
        "Find units that violate FAA regulations" → MUST be addressed

    Key differences:
        - Binary decision (compliant/non-compliant)
        - No cost tradeoff allowed
        - Regulatory thresholds are hard limits
        - Documentation required for audits

    Example:
        Turbine vibration exceeds ISO 10816 limit
        → MUST shut down, regardless of RUL or cost
        → Safety first

SECTION 6: EVALUATION METRICS QUESTIONS
================================================================================

Q6.1: How do you evaluate if an AI agent succeeded?
ROOT CAUSE: What makes a "good" answer?
SIMPLE ANSWER:
    Two-part evaluation:

    Part 1 - Correctness (Did it get the right answer?)
        - Compare AI output to ground truth
        - For RUL: Check MAE, RMSE within acceptable range
        - For Classification: Check accuracy, F1 score
        - For Cost: Check if savings calculated correctly

    Part 2 - Completeness (Did it provide everything required?)
        - Check all required_fields are present
        - Check output matches output_template format
        - Check ground_truth_verification was performed

    Both must pass to succeed

Q6.2: What does MAE tell you that RMSE doesn't?
ROOT CAUSE: Why measure error two ways?
SIMPLE ANSWER:
    MAE (Mean Absolute Error):
        - Average distance from correct answer
        - Treats all errors equally
        - Example: Errors of [5, 5, 5] → MAE = 5
        - Good for: Understanding typical error

    RMSE (Root Mean Squared Error):
        - Penalizes large errors more heavily
        - Squares errors before averaging
        - Example: Errors of [5, 5, 5] → RMSE = 5
                  Errors of [1, 1, 13] → MAE = 5, but RMSE = 7.6
        - Good for: Catching dangerous outliers

    In maintenance:
        - Small MAE = Generally accurate
        - Large RMSE = Has some very wrong predictions (risky!)
        - You want BOTH to be low

Q6.3: What's the difference between accuracy and F1 score?
ROOT CAUSE: Why not just use accuracy?
SIMPLE ANSWER:
    Accuracy can be misleading with imbalanced data:

    Example: 100 bearings, 95 normal, 5 faulty

    Dumb model that always says "normal":
        Accuracy = 95% (seems great!)
        But: Missed ALL 5 failures (disaster!)

    F1 Score balances:
        - Precision: Of predicted faults, how many were real?
        - Recall: Of real faults, how many did we catch?

    Same dumb model:
        F1 Score = 0% (correctly shows it's useless for faults)

    For safety: F1 > Accuracy (missing a fault is expensive/dangerous)

Q6.4: What does ground_truth_verification_status mean?
ROOT CAUSE: How do we know ground truth is correct?
SIMPLE ANSWER:
    It's a verification step that confirms:

    1. Ground truth data exists
    2. Ground truth data is accessible
    3. Ground truth matches expected format
    4. Agent's predictions can be compared to ground truth

    Example verification checks:
        ✓ RUL_FD001.txt file exists
        ✓ File has 100 lines (100 test units)
        ✓ All values are numeric
        ✓ Agent's predicted units are valid IDs (0-99)

    Status = True → Safe to evaluate
    Status = False → Cannot trust evaluation

SECTION 7: FUZZY DESCRIPTION QUESTIONS
================================================================================

Q7.1: What is fuzzy_description and why "fuzzy"?
ROOT CAUSE: Why not just give precise instructions?
SIMPLE ANSWER:
    Real-world tasks are described in natural language, not code:

    Precise (technical):
        "Train LSTM model on CMAPSS FD001, predict RUL for test set,
         return top 10 units with lowest predicted RUL"

    Fuzzy (how humans actually talk):
        "We need to identify which turbine engines are most likely to
         fail soon. Look at the degradation patterns and tell me the top 10
         at-risk units that need immediate attention."

    Why fuzzy?
        - Tests if AI can understand ambiguous requests
        - Mirrors real factory floor communication
        - Requires AI to infer intent, not just follow commands
        - More realistic evaluation

Q7.2: What makes a good fuzzy description?
ROOT CAUSE: What information must be included?
SIMPLE ANSWER:
    Must include (implicitly or explicitly):

    1. The Goal:
       "Identify engines that will fail soon"

    2. The Context:
       "We have sensor data from 100 turbines"

    3. The Output Needed:
       "Top 10 at-risk units"

    4. The Constraints:
       "Use historical degradation patterns"
       "We need this for tomorrow's maintenance planning"

    5. Why It Matters:
       "To prevent unexpected failures during flight"

    Good: Natural language that maintenance engineers would actually use
    Bad: Pseudo-code or overly technical jargon

Q7.3: Why can't fuzzy descriptions have contractions?
ROOT CAUSE: Why did SMEs flag "I'm", "We've", etc.?
SIMPLE ANSWER:
    Professional consistency and clarity:

    Before (with contractions):
        "We've noticed some bearings aren't performing well.
         I'm concerned they'll fail soon."

    After (formal):
        "We have noticed some bearings are not performing well.
         I am concerned they will fail soon."

    Reasons:
        - Professional documentation standards
        - Easier for non-native English speakers
        - Clearer for AI parsing
        - Matches formal engineering reports
        - Consistent tone across all 75 scenarios

Q7.4: How specific should fuzzy descriptions be about metrics?
ROOT CAUSE: Should we mention MAE, RMSE explicitly?
SIMPLE ANSWER:
    General guidance, NOT specific numbers:

    Good (appropriate specificity):
        "Predict when each turbine will fail. Your predictions should
         be reasonably accurate - within industry-standard error margins."

    Too Vague:
        "Do some predictions."

    Too Specific:
        "Achieve MAE between 12.5 and 25.1 cycles with RMSE below 33.4"
        (This gives away the ground truth!)

    Rule: Describe the task naturally; ground_truth handles evaluation

    Low Complexity (3-4):
        - Single dataset
        - 1-2 tools needed
        - Clear, unambiguous goal

        - Multiple datasets/agents

Q8.3: Why list required_tools?

    Without tool list:

        AI knows what's available and expected
        Can validate if it's using appropriate tools

    Also helps evaluation:
        - Did AI use the right tools?
        - Did AI create new tools when needed?
        - Did AI skip essential steps?

    Think of it like a recipe listing "required equipment"

Q8.4: What if the AI needs a tool not in required_tools?
ROOT CAUSE: Is the list comprehensive or just minimum requirements?
SIMPLE ANSWER:
    Two types of tools:

    1. Required (must use):
       - Listed in dependency_analysis
       - Essential for task completion
       - Example: load_data, validate_predictions

    2. Optional (may create/use):
       - Not listed, but helpful
       - AI can create dynamically
       - Example: specialized visualization tool,
                  custom preprocessing function

    Evaluation:
        ✓ Used all required tools
        ✓ Created helpful additional tools
        ✗ Skipped required tools
        ✗ Created unnecessary tools (overcomplicated)

    Why strict?
        - Ensures consistent evaluation
        - Prevents ambiguous results
        - Enables automated processing
        - Matches real API contract requirements

        - Parallel execution (faster evaluation)

        Can run pdm_rul_001, pdm_fault_010, pdm_cost_003 simultaneously

Q9.4: How are multi-agent scenarios different?

        "Predict RUL for bearings"

Q11.1: What are common mistakes when using these scenarios?
ROOT CAUSE: What goes wrong most often?
SIMPLE ANSWER:

    Mistake 1: Using placeholder ground truth
        Wrong: Thinking [0, 1, 2, 3, 4] is actual data
        Right: These are real unit IDs from the dataset
        Fix: Always extract from actual dataset files

    Mistake 2: Wrong dataset for task type
        Wrong: Using CWRU (fault classification) for RUL prediction
        Right: Match dataset capabilities to task requirements
        Fix: Check dataset has required ground truth type

    Mistake 3: Ignoring output_template
        Wrong: Returning data in any format
        Right: Exact field names and data types as specified
        Fix: Validate output against template before submitting

    Mistake 4: MAE/RMSE for subset
        Wrong: Calculating metrics only for top N units
        Right: Calculate across ALL test units
        Fix: Use complete test set, not just top predictions

    Mistake 5: Skipping ground_truth_verification
        Wrong: Assuming ground truth is valid
        Right: Verify accessibility and format
        Fix: Check files exist and are readable before evaluation

Q11.2: How do you debug a failing scenario?
ROOT CAUSE: Where to start when something goes wrong?
SIMPLE ANSWER:

    Step 1: Check Data Access
        - Can you load the dataset?
        - Does the file exist at expected path?
        - Are file permissions correct?
        Command: Check DATA_DIR in scenario

    Step 2: Verify Ground Truth
        - Does ground_truth have sample_correct_units?
        - Do the unit IDs exist in dataset?
        - Is expected_top_units_count correct?
        Command: Print len(sample_correct_units)

    Step 3: Validate Output Format
        - Does output match output_template?
        - Are all required_fields present?
        - Are data types correct?
        Command: Compare dict.keys() to required_fields

    Step 4: Check Calculations
        - Is MAE calculated correctly?
        - Is RMSE formula right?
        - Are you using ALL test units?
        Command: Print intermediate calculation steps

    Step 5: Review Logs
        - What tools were called?
        - What errors occurred?
        - What was the agent's reasoning?
        Command: Check execution logs/trajectories

Q11.3: What if ground truth seems wrong?
ROOT CAUSE: How to verify if ground truth is actually correct?
SIMPLE ANSWER:

    Verification steps:

    1. Check Source File:
       - For CMAPSS: RUL_FD001.txt should have ~100 lines
       - For FEMTO: CSV should have rul_percentage column
       - For CWRU: CSV should have label column

    2. Verify Extraction:
       - Re-run extract_ground_truth.py
       - Check logs for extraction process
       - Compare before/after values

    3. Manual Spot Check:
       - Pick a sample_correct_unit ID
       - Look up actual data for that unit
       - Confirm it matches expected characteristics
       Example: Unit 34 in CMAPSS_FD001 should have low RUL

    4. Statistical Validation:
       - MAE/RMSE ranges should align with literature
       - Values should be physically plausible
       - Distribution should match dataset characteristics

    5. Consult SMEs:
       - Do values match industry experience?
       - Are thresholds reasonable?
       - Would this be useful in practice?

Q12.1: How do scenarios support transfer learning?
ROOT CAUSE: Can knowledge from one scenario help with others?
SIMPLE ANSWER:

    Scenarios are grouped for knowledge transfer:

    Same Classification Type:
        - All RUL scenarios share prediction techniques
        - All Fault scenarios share classification approaches
        - Agent can learn patterns across similar tasks

    Same Dataset:
        - Multiple scenarios use CMAPSS_FD001
        - Agent learns dataset characteristics
        - Preprocessing becomes reusable

    Similar Complexity:
        - Complexity 3-4 scenarios teach basic workflows
        - Complexity 7-8 scenarios build on basics
        - Progressive difficulty enables learning

    Example Transfer:
        Master pdm_rul_001 (CMAPSS_FD001) →
        → Easier to solve pdm_rul_004 (CMAPSS_FD001, different goal)
        → Can transfer to pdm_rul_002 (CMAPSS_FD002, similar dataset)

Q12.2: How do you add new scenarios?
ROOT CAUSE: What's the process for extending the benchmark?
SIMPLE ANSWER:

    Step 1: Define Task
        - Identify real-world maintenance need
        - Determine classification_type
        - Set complexity_rating

    Step 2: Select Dataset
        - Choose dataset that has required ground truth
        - Verify dataset supports task type
        - Confirm data accessibility

    Step 3: Extract Ground Truth
        - Run extraction script on dataset
        - Verify sample_correct_units are valid
        - Calculate MAE/RMSE ranges if applicable

    Step 4: Write Scenario
        - Create fuzzy_description (natural language)
        - Define required_tools and dependencies
        - Specify output_template and required_fields

    Step 5: SME Validation
        - Have maintenance expert review
        - Incorporate feedback
        - Verify practical relevance

    Step 6: Test
        - Run with reference implementation
        - Verify evaluation works correctly
        - Document any edge cases

Q12.3: What makes a scenario "expert-vetted" vs just "technically correct"?
ROOT CAUSE: What's the difference in quality?
SIMPLE ANSWER:

    Technically Correct Scenario:
        - JSON is valid
        - Ground truth exists
        - Metrics are calculable
        - Code can execute it

        Example: "Predict RUL with MAE < 20"
        Problem: Is 20 the right threshold? Says who?

    Expert-Vetted Scenario:
        - Everything above, PLUS:
        - Reflects real maintenance workflows
        - Uses industry-standard metrics
        - Thresholds match practical experience
        - Language matches how engineers communicate

        Example: "Predict RUL with MAE in range [12.5, 25.1]
                  based on historical CMAPSS benchmark performance"
        Better: SME confirmed this range from actual deployment data

    Key Difference:
        Technical = Works in lab
        Expert-vetted = Works in factory

Q12.4: How do scenarios handle data drift?
ROOT CAUSE: What if real equipment data changes over time?
SIMPLE ANSWER:

    Current Approach (Static Benchmark):
        - Scenarios use fixed PDMBench datasets
        - Ground truth doesn't change
        - Ensures reproducible evaluation
        - Good for: Comparing AI models fairly

    Real-World Challenge:
        - Equipment ages differently
        - Operating conditions vary
        - Sensor accuracy degrades
        - Models need retraining

    Future Extension:
        - Add "data drift" scenarios
        - Include temporal validation
        - Test model robustness to changes

        Example:
            Scenario A: Train on 2020 data, test on 2020 data
            Scenario B: Train on 2020 data, test on 2023 data
            → Measures degradation of model performance

    For Now:
        Scenarios provide baseline evaluation
        Assume data distribution is stable
        Focus on initial model performance
